# AI_RULES.md

## 1. Назначение документа

Этот документ фиксирует правила использования AI-инструментов при работе над проектом.

Его задача — сделать так, чтобы любые будущие сессии, промпты и генерации кода оставались в рамках уже утверждённого направления проекта и не ломали:

- концепт игры
- структуру первого vertical slice
- выбранный стек
- утверждённые рамки MVP
- логику сцен
- тон и атмосферу проекта

Этот документ нужен как ограничитель и ориентир одновременно.

---

## 2. Базовый контекст проекта

Проект — это короткая браузерная 2D point-and-click игра в готическом пиксельном стиле, сделанная как промо-экспириенс для музыкального альбома.

### Основные принципы проекта

- игра короткая
- игра строится вокруг атмосферы, а не высокой сложности
- проект состоит из глав
- каждая глава связана с отдельным треком
- каждая глава включает дневную сцену и ночную мини-игру
- убийства не показываются напрямую до финала
- игра не должна превращаться в детектив
- текущий приоритет — собрать первый рабочий vertical slice

AI не должен предлагать решения, которые противоречат этим базовым принципам.

---

## 3. Текущий этап проекта

AI должен ориентироваться на текущий приоритетный этап разработки и в первую очередь помогать с теми задачами, которые напрямую двигают проект к ближайшему рабочему результату.

На текущем этапе таким приоритетом является первый playable vertical slice, зафиксированный в документах:

- `README.md`
- `PRD_V1`
- `LEVEL_01_V1.md`
- `TECH_PLAN_V1.md`
- `AI_RULES.md`

Пока первый vertical slice не собран и не проходит от начала до конца, AI не должен преждевременно уводить проект в сторону:

- полной архитектуры всей игры
- сложной системы контента для всех глав
- масштабных универсальных решений «на будущее»
- второстепенных систем, не нужных для первого рабочего прототипа

---

## 4. Утверждённый стек

AI должен опираться только на следующий стек:

- **TypeScript**
- **Node.js**
- **Phaser**
- **Vite**

### AI не должен предлагать вместо этого

- Next.js
- React как основную платформу игры
- backend без явной необходимости
- database
- Electron
- Unity
- Unreal
- Godot
- тяжёлую framework-архитектуру

Если пользователь прямо не попросил поменять стек, AI обязан придерживаться текущего выбора.

---

## 5. Приоритет MVP над масштабируемостью

Главное правило текущего этапа:

**сначала рабочий vertical slice, потом масштабирование**

### Это означает

AI должен предпочитать:

- простые решения
- минимальные зависимости
- понятную структуру
- явные флаги состояния
- маленькие локальные системы
- код, который легко прочитать и быстро исправить

AI не должен без необходимости предлагать:

- сложные абстракции
- обобщённые framework-like решения
- ECS-архитектуру
- event bus «на будущее»
- универсальные менеджеры всего
- сложную dependency injection схему
- многоуровневую data-driven архитектуру до появления реальной необходимости

---

## 6. Правила по структуре кода

AI должен писать код так, чтобы он был:

- читаемым
- прямолинейным
- коротким
- модульным
- удобным для ручного редактирования
- реалистичным для первого прототипа

### Предпочтительно

- небольшие файлы
- явные названия
- минимум магии
- логика сцены рядом с логикой сцены
- данные уровня отдельно от общей системы, если это упрощает работу

### Нежелательно

- чрезмерная универсализация
- слишком раннее разбиение на десятки абстрактных слоёв
- «идеальная архитектура» без реальной необходимости
- код, который сложно дебажить
- скрытая логика, завязанная на сложные цепочки вызовов

---

## 7. Правила по документации

Если AI помогает писать документы для репозитория, он должен:

- сохранять единый язык документации
- придерживаться уже принятой структуры файлов
- не смешивать русский и английский без причины
- не ломать markdown-формат
- отдавать документы в виде, пригодном для прямой вставки в `.md`

### Для текущего проекта

На текущем этапе документация ведётся **на русском языке**.

Следовательно:

- новые документы должны по умолчанию быть на русском
- commit message и описания можно делать на русском, если это удобно пользователю
- технические названия файлов, классов и переменных могут оставаться на английском

---

## 8. Правило формата документов

Если AI пишет документ для репозитория, он должен отдавать его:

- в чистом markdown-формате
- без сломанной вложенности code block
- без служебных строк
- без случайных технических вставок
- в виде, который можно сразу скопировать в `.md` файл

AI не должен оформлять документ так, чтобы после копирования в репозиторий ломались заголовки, списки или кодовые блоки.

---

## 9. Правила по сюжетной согласованности

AI не должен предлагать изменения, которые ломают уже утверждённую сюжетную рамку проекта.

### Уже зафиксировано

- герой — бедный молодой приезжий из деревни
- он приезжает в большой готический город
- он хочет пробиться вверх в музыкальном мире города
- каждая глава связана с женщиной, которая помогает ему продвинуться дальше
- ночные мини-игры только в финале переосмысляются как реальные убийства
- убийства не показываются напрямую в процессе игры
- последствия пропаж должны быть тонкими, а не детективными

### Следовательно

AI не должен предлагать:

- прямой показ убийства в первых уровнях
- раннее раскрытие правды
- полицейское расследование как ведущую механику
- боевую систему
- радикальную смену сеттинга
- современный UI или вайб, противоречащий готическому миру
- полную жанровую смену проекта

---

## 10. Правила по утверждённым уровням

Если уровень уже зафиксирован отдельным документом, AI не должен без причины пересобирать его базовый flow, сцены и ключевые события.

Изменения допустимы только если:

- пользователь явно просит пересмотр
- найдено противоречие с `PRD_V1`
- требуется локальное улучшение без разрушения основы

### Отдельно для `LEVEL_01_V1.md`

`LEVEL_01_V1.md` является первым утверждённым vertical slice проекта.

AI должен считать его текущую базовую структуру зафиксированной и не пересобирать её без явной причины или нового решения пользователя.

К зафиксированным элементам относятся:

- узкая торговая улица
- девушка у сундука
- короткий диалог с выбором реплики
- поиск тонкого металлического предмета / шпильки
- короткая мини-механика вскрытия сундука
- деньги на ночлег
- дешёвая комната
- переход ко сну
- первая ночная сцена / заглушка мини-игры
- отсутствие прямого показа убийства

---

## 11. Правила по игровым системам

На этапе первого прототипа AI должен опираться на следующие базовые системы:

- Dialogue system
- Interaction system
- Transition system
- Level state flags

### AI не должен навязывать раньше времени

- полноценный inventory system
- глобальный quest manager
- сложный cutscene framework
- save/load architecture
- dialogue editor
- level editor
- achievements
- map system
- complex combat / stealth / RPG systems

---

## 12. Правила по UI

UI первого прототипа должен быть:

- минимальным
- читаемым
- атмосферным
- неброским
- вторичным по отношению к сцене

AI должен предлагать простой UI для:

- текста диалогов
- имени говорящего
- выбора реплик
- минимальной lockpick-механики
- мягких переходов между сценами

AI не должен предлагать тяжёлый, перегруженный или слишком «игровой» интерфейс, если в этом нет необходимости.

---

## 13. Правила по анимации и визуальной подаче

AI должен помнить, что проект не строится вокруг сложной анимации на старте.

### Допустимо предлагать

- idle-анимации
- walk-анимации
- blink-анимации
- flicker свечей и фонарей
- fade in / fade out
- tween-based UI-эффекты
- простые атмосферные переходы

### Не нужно предлагать на старте

- сложный skeletal animation pipeline
- дорогую покадровую анимацию всего подряд
- cinematic-heavy production scope
- длинные cutscene-секвенции

---

## 14. Правила по мини-играм

AI должен помнить:

- мини-игры должны быть короткими
- понятными
- разными
- не случайными по смыслу
- связанными с темами подавления, стирания, завершения, гашения и ритуального действия

### Для текущего этапа

В `Level 01` допустима простая заглушка или proof-of-structure версия мини-игры.

AI не должен настаивать на полной финальной реализации всей системы мини-игр до того, как заработает основной flow уровня.

---

## 15. Правила по генерации кода

Когда AI пишет код для проекта, он должен:

- использовать TypeScript
- учитывать Phaser-структуру
- делать код пригодным для вставки в существующий проект
- писать имена классов, методов и файлов последовательно
- избегать лишней сложности
- добавлять только ту инфраструктуру, которая реально используется

### Если AI пишет новый модуль

Он должен чётко понимать:

- где этот модуль будет лежать
- кто его вызывает
- какую задачу он решает
- нужен ли он вообще на этапе MVP

---

## 16. Правило внесения изменений

Если AI предлагает изменить уже утверждённый документ, сцену, механику или решение, он должен:

1. явно указать, что именно предлагается поменять
2. объяснить, зачем это изменение нужно
3. не подменять утверждённую логику без явного согласования
4. не заменять уже принятые решения просто потому, что можно придумать другой вариант

### Предпочтительный стиль

Не перепридумывать весь проект, а улучшать:

- локально
- аргументированно
- экономно
- без разрушения уже собранного фундамента

---

## 17. Правила по работе с пользователем

AI должен помнить, что пользователь хочет:

- быстро двигаться вперёд
- не тонуть в лишней теории
- получать решения, пригодные для практики
- собирать проект постепенно
- фиксировать результат документами
- не расползаться по бесконечным вариантам

Следовательно, AI должен:

- быть конкретным
- не растекаться по лишним альтернативам
- не навязывать переусложнение
- предлагать следующий понятный шаг
- помогать доводить документы и код до состояния «можно брать и вставлять»

---

## 18. Что AI должен делать при неопределённости

Если какая-то часть ещё не зафиксирована, AI должен:

- опираться на уже утверждённые документы
- предлагать решение, которое лучше всего подходит под MVP
- не ломать общий тон проекта
- выбирать простейший жизнеспособный вариант

### Предпочтение по умолчанию

Если есть выбор между:

- красивой сложной системой
- и простой рабочей системой

AI должен выбирать **простую рабочую систему**, если пользователь явно не попросил иного.

---

## 19. Что AI не должен делать

AI не должен:

- забывать утверждённый стек
- смешивать проект игры с веб-приложением на React/Next
- преждевременно расширять scope
- переписывать narrative-ядро проекта
- делать документацию в ломаном формате
- генерировать код в вакууме без связи с текущей структурой
- предлагать решения, которые плохо подходят для первого прототипа
- путать атмосферный point-and-click проект с системной sandbox-игрой

---

## 20. Приоритет источников истины

Если возникает конфликт между перепиской, документами и более ранними черновиками, приоритет должен быть таким:

1. последние явно утверждённые решения пользователя
2. `LEVEL_01_V1.md` для первого уровня
3. `PRD_V1` для общей концепции проекта
4. `TECH_PLAN_V1.md` для технической реализации
5. `AI_RULES.md` как ограничивающий документ для AI
6. более ранние черновики и старые версии

AI должен относиться к репозиторию и утверждённым документам как к главной рабочей базе проекта.

---

## 21. Краткое резюме

`AI_RULES.md` нужен, чтобы все следующие сессии и генерации оставались в рамках проекта, а не тянули его в случайные стороны.

Главное правило:

**не строить лишнее, не ломать утверждённое, двигать проект к рабочему vertical slice**

На текущем этапе AI должен помогать прежде всего в следующем:

- документация
- структура проекта
- базовые системы
- сцены первого уровня
- простой и чистый код
- MVP-реализация без оверинжиниринга
