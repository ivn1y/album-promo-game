# TECH_PLAN_V2.md

## Проект
**album-promo-game**

## Технологический стек
- TypeScript
- Phaser
- Vite
- Node.js

Игра запускается как браузерное приложение и разрабатывается как небольшой scene-based проект без backend-части.

---

## 1. Назначение документа

Этот документ фиксирует техническую структуру проекта и базовые принципы реализации.

Цель документа:
- определить минимально необходимую архитектуру
- зафиксировать базовые системы игры
- не допустить оверинжиниринга
- сохранить фокус на playable vertical slice

Документ не описывает сюжет по сценам и не заменяет PRD.  
Он нужен для того, чтобы кодовая структура проекта оставалась простой, понятной и соответствовала текущему этапу разработки.

---

## 2. Общий технический подход

Проект строится как небольшая 2D point-and-click игра на Phaser.

Основной принцип разработки:

**сначала рабочая логика → потом визуальный слой → потом замена ассетов → потом полировка**

На текущем этапе проекту не нужны:
- backend
- база данных
- сложная ECS-архитектура
- тяжёлые менеджеры состояний
- большие UI-системы
- сложный инвентарь
- продвинутая система дедукции
- лишняя абстракция “на будущее”

Архитектура должна быть:
- простой
- сцено-ориентированной
- легко читаемой
- удобной для постепенного расширения

---

## 3. Основной игровой цикл

Игра строится на повторяющемся цикле:

1. дневная сцена расследования
2. поиск улик
3. короткий вывод героя
4. переход к ночной сцене
5. ночная мини-игра
6. определение результата ночи
7. изменение состояния мира
8. запуск следующего кейса

Это ядро всей игры.

Именно этот цикл должен быть подтверждён первым рабочим vertical slice.

---

## 4. Система сцен

Игра должна быть разбита на небольшие независимые сцены Phaser.

Минимальный набор сцен для текущей архитектуры:

### BootScene
Отвечает за:
- загрузку базовых ассетов
- старт игры
- переход в первую игровую сцену

### DayCaseScene
Основная дневная сцена расследования.

Отвечает за:
- отображение локации
- интерактивные точки
- NPC
- диалоги
- поиск улик
- запуск вывода героя
- переход к ночной сцене

### NightMiniGameScene
Ночная мини-игра.

Отвечает за:
- запуск короткой абстрактной ночной сцены
- фиксацию результата: success / fail / skip
- передачу результата в глобальное состояние игры

### TransitionScene (опционально)
Используется для:
- коротких текстовых вставок
- fade in / fade out
- смены времени суток
- атмосферных переходов между сценами

### EndingScene
Финальная сцена.

Используется для:
- основного финала
- альтернативного финала без убийств
- финальных текстов / визуала / музыки

---

## 5. Дневная сцена расследования

Дневная сцена — основной игровой экран.

Она должна содержать:
- фон локации
- интерактивные хотспоты
- NPC
- систему диалогов
- систему поиска улик

Игрок в этой сцене должен уметь:
- перемещать внимание по точкам интереса
- осматривать окружение
- кликать по объектам
- разговаривать с персонажами
- находить улики
- собирать достаточный объём информации для короткого вывода героя

Типовая структура дневной сцены:
1. игрок попадает на место события
2. получает краткий контекст
3. осматривает сцену
4. находит 2–3 ключевые улики
5. запускает короткий вывод героя
6. сцена переходит к ночной части

---

## 6. Ночная мини-игра

Ночная сцена — короткий самостоятельный игровой эпизод.

Её задача:
- создать тревожный и сюрреалистичный контраст с дневной частью
- быть механически связанной с убийством
- не раскрывать эту связь напрямую до финала

Ночная мини-игра должна восприниматься как:
- сон
- ритуал
- искажённое внутреннее действие
- психологическая сцена

Нельзя прямо показывать:
- сам акт убийства
- жертву как однозначную цель игрока
- прямое признание того, что герой — убийца

Допустимые результаты ночной сцены:
- `success`
- `fail`
- `skip`

Связь с дневной игрой:
- `success` → ночью произошло убийство
- `fail` или `skip` → убийства не произошло

---

## 7. Глобальное состояние игры

Проекту нужен простой глобальный state без лишней сложности.

Для этого достаточно одного централизованного объекта состояния, например `GameState`.

На текущем этапе в нём должны храниться только действительно важные данные.

### Пример ключевых полей

#### `murderCount`
Количество произошедших убийств.

Используется для:
- нарративного прогресса
- проверки ветки финала
- изменения диалогов
- роста напряжения

#### `cityMoodLevel`
Состояние визуального восприятия города.

Это не “цветастость”, а контролируемая шкала психологического состояния мира.

Например:
- `0` — базовое состояние
- `1` — слегка тусклее
- `2` — заметно серее
- `3` — тяжёлое выцветшее состояние

Важно:
- изменение должно быть постепенным
- игра не должна становиться слишком яркой
- серия убийств не делает мир “весёлым”
- серия пропущенных мини-игр делает мир заметно более серым и глухим

#### `nightMiniGameResult`
Результат последней ночной сцены.

Возможные значения:
- `success`
- `fail`
- `skip`

#### `currentCaseType`
Тип текущего или следующего расследования.

Возможные значения:
- `murder_case`
- `normal_case`

#### `caseIndex`
Порядковый номер текущего кейса / главы.

Используется для:
- линейного прогресса
- подбора нужной сцены
- контроля темпа игры

---

## 8. Логика выбора следующего кейса

После каждой ночной сцены игра должна определить, какое дело получит игрок следующим днём.

Базовый принцип:

- если `nightMiniGameResult = success` → следующий кейс `murder_case`
- если `nightMiniGameResult = fail` или `skip` → следующий кейс `normal_case`

При этом:
- игра остаётся линейной
- структура сцены остаётся похожей
- меняется нарративный контекст, атмосфера и содержание расследования

То есть технически система не должна превращаться в сложную нелинейную ветвящуюся сеть.

Нужна не полноценная детективная песочница, а управляемая последовательность эпизодов.

---

## 9. Система визуального состояния города

Игра должна поддерживать постепенное изменение визуального состояния мира.

Это не отдельная “большая графическая система”, а простой управляемый параметр, влияющий на подачу сцены.

Изменяться могут:
- насыщенность
- контраст
- тепло / холод освещения
- количество дымки / тумана
- общая читаемость сцены

### Главный принцип
Изменение визуала должно быть:
- постепенным
- ограниченным
- психологическим, а не карикатурным

Что важно:
- игра не должна становиться яркой и насыщенной после каждого убийства
- один успешный эпизод не должен резко “оживлять” мир
- более сильное направление изменения — это постепенное умирание и выцветание мира при отсутствии убийств
- убийство может лишь слегка сдвинуть сцену обратно от полной серости

Технически это можно реализовать через:
- несколько подготовленных версий фона
- небольшой набор пост-эффектов
- простой параметр mood level, который влияет на подачу сцены

На текущем этапе не нужно строить тяжёлую универсальную систему постобработки.

---

## 10. NPC-система

NPC нужны для:
- выдачи контекста
- коротких диалогов
- атмосферы
- сопровождения расследования

На текущем этапе NPC не требуют сложного поведения.

Минимально достаточно:
- позиции на сцене
- интерактивности по клику
- короткого набора реплик
- простой реакции на состояние кейса или города

Особенно важен повторяющийся NPC-помощник.

Помощник нужен для:
- подачи информации
- структурирования кейсов
- реплик-контраста к детективу
- постепенного внешнего взгляда на странности героя

---

## 11. Система улик

Каждая дневная сцена должна содержать ограниченное число интерактивных точек.

Примеры улик:
- след обуви
- сломанный замок
- отпечатки
- кусок ткани
- кровь
- царапина
- след взлома
- забытый предмет

Важно:
- не перегружать сцену
- делать 2–3 действительно значимые улики
- остальные точки могут работать как атмосфера или дополнительные наблюдения

После нахождения ключевого набора улик должен активироваться короткий вывод героя.

Проекту пока не нужна сложная система хранения, сопоставления и ручного комбинирования улик.

---

## 12. Система диалогов

Диалоги должны оставаться простыми и управляемыми.

На текущем этапе достаточно:
- последовательного показа реплик
- простых выборов в отдельных сценах
- запуска диалога по интеракции
- завершения диалога с изменением флагов сцены

Диалоговая система должна уметь:
- блокировать ввод на время разговора
- запускать события после завершения реплики
- поддерживать короткие ветки выбора там, где это нужно

Не нужно строить тяжёлую универсальную narrative-систему.

---

## 13. Система интеракций

Все основные действия игрока строятся на point-and-click логике.

Нужны:
- интерактивные зоны
- реакция на hover / click
- проверка условий для доступа к действию
- простой вызов диалога, осмотра или события

Интерактивные элементы могут быть:
- невидимыми хотспотами
- лёгкими debug-зонами на этапе разработки
- отдельными объектами поверх фона

Важно не смешивать:
- фон сцены
- интерактивные точки
- персонажей
- UI

Это упростит замену визуала в будущем.

---

## 14. UI

Проекту нужен минимальный интерфейс.

На текущем этапе достаточно:
- окна диалога
- простого текста осмотра
- лёгкой подсветки интерактивных зон
- минимальных переходных надписей

Не нужны:
- полноценный инвентарь
- журнал квестов
- большой HUD
- перегруженные меню

UI должен быть:
- минимальным
- читаемым
- атмосферным
- не выбивающимся из общей стилистики

## 16. Vertical Slice

Первый рабочий vertical slice должен доказать, что базовое ядро игры работает.

Минимальный состав vertical slice:
- 1 дневная сцена расследования
- 1 ночная мини-игра
- 1 результат ночи
- 1 изменение состояния мира
- 1 следующий тип кейса в зависимости от результата

Если этот цикл работает, значит архитектура проекта выбрана правильно.

---

## 17. Порядок разработки

Разработка должна идти в таком порядке:

1. базовая сцена и переходы
2. point-and-click интеракции
3. сбор улик и вывод героя
4. ночная мини-игра
5. глобальное состояние игры
6. зависимость следующего кейса от результата ночи
7. визуальное состояние города
8. замена заглушек на реальные ассеты
9. полировка

Нельзя:
- сначала строить большую архитектуру
- сначала делать всю графику
- сразу закладывать сложную вариативность
- расползаться в системы до подтверждения core loop

---

## 18. Технический принцип проекта

Главная техническая цель на текущем этапе — не “идеальная архитектура”, а **маленький, чистый и устойчивый игровой каркас**, который:

- запускается локально без лишней сложности
- легко расширяется на 1–2 следующих кейса
- не ломается от смены визуала
- не требует переписывания половины проекта при каждом изменении сценария

Если при разработке возникает выбор между:
- “более умной архитектурой”
- и “более простой рабочей реализацией”

предпочтение нужно отдавать **более простой рабочей реализации**.
