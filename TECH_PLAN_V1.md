# TECH_PLAN.md

## 1. Назначение документа

Этот документ описывает технический план первого vertical slice проекта.

Его задача — перевести концепт из `PRD.md` и специфику первого уровня из `LEVEL_01.md` в понятную техническую структуру, на основе которой можно начинать разработку без лишних решений на ходу.

Документ фокусируется на:

- MVP-архитектуре
- структуре проекта
- ключевых сценах
- базовых системах
- состоянии уровня
- порядке реализации
- технических ограничениях первого прототипа

---

## 2. Цель первого technical prototype

Первый technical prototype должен доказать, что в проекте уже работает базовый игровой цикл:

- загрузка игры
- запуск первой сцены
- взаимодействие с интерактивными объектами
- запуск диалога
- выбор реплики
- поиск предмета
- открытие сундука через короткую механику
- получение награды
- переход в комнату
- взаимодействие с кроватью
- переход в сон
- запуск заглушки ночной мини-игры

На этом этапе важна не полировка, а работающий vertical slice.

---

## 3. Стек

Проект собирается на следующем стеке:

- **TypeScript**
- **Node.js**
- **Phaser**
- **Vite**

### Что это значит на практике

- `Node.js` нужен для окружения, npm и запуска проекта
- `Vite` нужен для dev-сервера и сборки
- `TypeScript` используется как основной язык
- `Phaser` — основной игровой фреймворк

### Что не используется

- `Next.js`
- `React`
- backend
- база данных
- серверная логика

Проект должен быть простым клиентским браузерным приложением.

---

## 4. Технические цели MVP

Для первого MVP должны быть реализованы только те вещи, без которых невозможно пройти первый уровень от начала до конца.

### Обязательно нужно

- запуск проекта локально
- одна рабочая конфигурация Phaser
- сцена улицы
- сцена комнаты
- сцена сна / заглушка мини-игры
- базовая система диалогов
- базовая система интеракций
- флаги состояния уровня
- короткие переходы между сценами
- простая логика пазла
- короткая мини-механика вскрытия сундука
- минимальный UI для текста / выбора

### Пока не нужно

- общая карта города
- глобальная сюжетная система
- инвентарь в полном виде
- сохранения
- настройки
- сложный UI
- полноценная система квестов
- полная система катсцен
- сложное перемещение по локации
- полировка анимации
- финальная версия мини-игр

---

## 5. Общий подход к архитектуре

Для первого vertical slice не нужна большая масштабная архитектура.

Подход должен быть:

- простым
- модульным
- понятным
- легко расширяемым
- без оверинжиниринга

### Главный принцип

Сначала собирается **рабочий скелет игры**, а не «идеальная архитектура на будущее».

То есть:

- лучше несколько простых модулей, чем сложная абстракция
- лучше явные state flags, чем ранняя универсальная система всего
- лучше простой scene flow, чем перегруженная глобальная orchestration-схема

## 7. Роли ключевых файлов

### `main.ts`

Точка входа приложения.

Должен:
- создавать экземпляр Phaser Game
- подключать `gameConfig`

### `game/config/gameConfig.ts`

Основной конфиг Phaser.

Должен содержать:
- размеры игры
- список сцен
- фоновые настройки
- тип рендера
- базовые параметры scale

### `BootScene.ts`

Стартовая техническая сцена.

Её задача:
- предзагрузка минимальных ассетов
- инициализация
- переход в `ArrivalStreetScene`

Для первого прототипа может быть очень простой.

### `ArrivalStreetScene.ts`

Главная сцена первого уровня.

Отвечает за:
- лэйаут улицы
- интерактивные объекты
- диалог с девушкой
- выбор реплики
- поиск шпильки / металлического предмета
- запуск мини-механики вскрытия
- открытие сундука
- финальный диалог
- переход в комнату

### `BoardingRoomScene.ts`

Сцена дешёвой комнаты.

Отвечает за:
- отображение комнаты
- короткие интеракции
- взаимодействие с кроватью
- запуск перехода ко сну

### `DreamIntroScene.ts`

Сцена сна / заглушка ночной мини-игры.

Отвечает за:
- затемнение и вход в ночной слой
- простую заглушку ночной механики
- завершение первого уровня

---

## 8. Flow сцен

Минимальный flow должен быть таким:

1. `BootScene`
2. `ArrivalStreetScene`
3. `BoardingRoomScene`
4. `DreamIntroScene`

### Логика переходов

- `BootScene` сразу передаёт управление в `ArrivalStreetScene`
- `ArrivalStreetScene` после завершения пазла и награды переводит игрока в `BoardingRoomScene`
- `BoardingRoomScene` после взаимодействия с кроватью переводит игрока в `DreamIntroScene`
- `DreamIntroScene` завершает vertical slice

---

## 9. Модель состояния уровня

Для первого уровня лучше использовать простую флаговую модель состояния.

### Почему так

Потому что:
- логика линейная
- сцен немного
- сложных ветвлений нет
- это быстрее в реализации и проще в отладке

### Пример состояния

```ts
type Level01State = {
  hasTalkedToGirl: boolean;
  hasChosenReply: boolean;
  selectedReplyTone: 1 | 2 | 3 | null;
  hasFoundTool: boolean;
  lockpickStarted: boolean;
  chestOpened: boolean;
  receivedMoney: boolean;
  enteredBoardingRoom: boolean;
  canSleep: boolean;
  dreamStarted: boolean;
  dreamCompleted: boolean;
};
```

### Где хранить

На этапе MVP состояние можно хранить:
- либо в отдельном простом объекте
- либо в маленьком state-модуле
- либо внутри scene + shared state object

Лучший вариант для старта:
- отдельный файл `levelState.ts`
- с экспортируемым объектом состояния и функцией сброса

## 11. Dialogue system

### Назначение

Dialogue system должна решать следующие задачи:

- показывать последовательность реплик
- поддерживать диалог между персонажами
- поддерживать выбор реплики игрока
- вызывать последствия выбора
- запускать следующий шаг сценария

### Для MVP достаточно

- текстового окна
- имени говорящего
- кнопки / клика для продолжения
- блока выбора из 3 реплик
- callback после завершения диалога

### Что не нужно пока

- ветвящееся дерево диалогов общего назначения
- сложная cinematic-подача
- портреты персонажей
- озвучка
- история диалога

---

## 12. Рекомендуемая модель диалога

Лучше описывать диалоги данными, а не хардкодить длинные цепочки прямо в scene.

Например:

```ts
type DialogueLine = {
  speaker: "girl" | "hero";
  text: string;
};

type DialogueChoice = {
  id: 1 | 2 | 3;
  text: string;
  response: DialogueLine[];
  effect?: () => void;
};
```

### Что должно быть в данных Level 01

- стартовый диалог
- выбор игрока
- три ответа девушки
- реплика с просьбой помочь
- финальный диалог после открытия сундука

---

## 13. Interaction system

### Назначение

Interaction system должна обрабатывать:
- клики по хотспотам
- реакцию на осмотр объекта
- проверку флагов
- запуск диалогов
- запуск событий

### Типы интеракций для MVP

- осмотр объекта
- разговор с NPC
- подбор предмета
- использование найденного предмета на сундуке
- взаимодействие с кроватью

### Пример логики

- игрок кликает по девушке → запускается диалог
- игрок кликает по сундуку до нахождения шпильки → короткая реплика героя
- игрок кликает по спрятанной шпильке → ставится флаг `hasFoundTool = true`
- игрок кликает по сундуку после нахождения шпильки → запускается мини-механика вскрытия
- игрок кликает по кровати → запускается переход ко сну

---

## 14. Мини-механика вскрытия сундука

### Назначение

Это очень короткая отдельная механика внутри `ArrivalStreetScene`.

Она нужна, чтобы:
- сделать пазл менее плоским
- дать игроку ощущение ручного действия
- сделать открытие сундука более игровым

### Требования

- длительность 3–8 секунд в идеале
- понятна с первой попытки
- не раздражает
- не превращается в отдельную сложную игру
- не требует отдельной большой системы

### Технический подход

Для MVP лучше сделать максимально простую реализацию.

Подходящий вариант:
- открывается overlay / popup поверх сцены
- показывается простая шкала или индикатор
- игрок должен нажать / удержать в нужной зоне
- при успехе:
  - `lockpickStarted = true`
  - `chestOpened = true`
  - overlay закрывается
  - запускается финальный диалог

### Что не нужно

- сложная физика замка
- несколько штифтов
- псевдо-реалистичный взлом
- серия попыток с наказанием

## 15. Transition system

### Назначение

Transition system нужна для:
- fade in / fade out
- затемнения при смене сцены
- мягкого перехода в сон
- общего ощущения цельности сцен

### Для MVP достаточно

- чёрного прямоугольного overlay
- tween на alpha
- callback по завершению

### Основные переходы

- старт улицы
- выход с улицы в комнату
- затемнение ко сну
- переход в `DreamIntroScene`

## 16. UI для MVP

Минимальный UI должен включать:

- текстовое окно для реплик
- имя говорящего
- меню выбора реплик
- минимальный overlay для lockpick-механики
- возможно, короткий title card для новой сцены

### Важно

UI должен быть:
- простым
- читаемым
- не перегруженным
- атмосферным, но без сложной декоративности на старте

## 17. Данные первого уровня

Лучше вынести значимую часть данных Level 01 в отдельные файлы.

### Что можно хранить отдельно

- тексты диалога
- описания интерактивных объектов
- короткие реакции на хотспоты
- стартовые флаги
- конфиг хотспотов

### Пример

```text
data/
  level01/
    dialogue.ts
    interactions.ts
    flags.ts
```

### Зачем это нужно

- scene становится чище
- тексты не смешиваются с логикой
- проще править контент
- проще подключать нейронки / редактировать данные позже

---

## 18. Placeholder-first подход

Первый прототип должен собираться через placeholder assets.

### Это значит

Можно временно использовать:
- простые прямоугольники
- временные спрайты
- заглушки для персонажей
- текст вместо красивых табличек
- однотонные фоны
- условные иконки для интерактивов

### Почему это важно

Если начать с финальной графики, проект застрянет слишком рано.

Сначала нужно доказать:
- что работает flow
- что сцены переключаются
- что интеракции понятны
- что диалоговая логика не ломается

## 19. Подход к перемещению игрока

На этапе MVP есть два допустимых варианта:

### Вариант A — без свободного movement

Игрок просто кликает по объектам, не управляя персонажем напрямую.

Плюсы:
- быстрее
- проще
- меньше технических проблем
- лучше для первого vertical slice

### Вариант B — минимальное movement

Игрок кликает по объекту, и герой делает короткий подход к нему.

Плюсы:
- атмосфернее
- живее

Минусы:
- сложнее
- нужен pathfinding или хотя бы простая логика позиционирования

### Рекомендация

Для первого прототипа лучше выбрать **вариант A**:
- без полноценного movement
- с акцентом на клики по хотспотам
- максимум — простая иллюзия смещения / анимации

## 20. Анимация в MVP

### Что можно анимировать покадрово

- idle героя
- idle / blink девушки
- мерцание свечи / фонаря

### Что лучше анимировать кодом

- fade in / fade out
- появление текстового окна
- hover-подсветку
- затемнение экрана
- переход в сон
- лёгкие UI-эффекты

### Ограничение

Не нужно пытаться сразу строить сложную animation pipeline.

---

## 21. Аудио в MVP

Для первого прототипа аудио может быть минимальным.

### Достаточно

- один музыкальный трек для улицы
- ослабленная / тихая версия или другой слой для комнаты
- минимальный абстрактный звук для сна
- клик по UI
- короткий звук успеха при открытии сундука

### Не нужно пока

- сложный амбиент-микс
- десятки звуков окружения
- полноценный adaptive soundtrack

## 22. Порядок реализации

Рекомендуемый порядок разработки:

### Этап 1 — базовый каркас

3. собрать `gameConfig`
4. добавить `BootScene`
5. убедиться, что игра запускается

### Этап 2 — scene flow

6. добавить `ArrivalStreetScene`
7. добавить `BoardingRoomScene`
8. добавить `DreamIntroScene`
9. настроить переходы между сценами

### Этап 3 — интеракции

10. добавить хотспоты на улице
11. добавить девушку как интерактивный объект
12. добавить простые inspect-реакции
13. добавить спрятанную шпильку
14. добавить сундук с проверкой состояния

### Этап 4 — диалоги

15. реализовать систему реплик
16. добавить стартовый диалог
17. добавить выбор из 3 реплик

### Этап 5 — вскрытие сундука

20. сделать короткий overlay lockpick-механики

### Этап 6 — комната и сон

23. собрать сцену комнаты
24. добавить кровать как главную интеракцию

### Этап 7 — ночная заглушка

29. завершить уровень

---

## 23. Критерии готовности technical prototype

Technical prototype можно считать успешным, если:

- проект запускается одной командой
- сцены переключаются без ошибок
- первый диалог работает
- выбор реплики работает
- шпилька подбирается
- сундук открывается только после правильной цепочки
- reward-диалог запускается корректно
- переход в комнату работает
- кровать запускает сон
- dream scene открывается без поломок

---

## 24. Что не надо делать раньше времени

На этом этапе не стоит:

- строить «универсальную систему для всей игры»
- делать глобальный quest manager
- придумывать сложную ECS-архитектуру
- тащить тяжёлые UI-библиотеки
- делать полноценный инвентарь
- писать сложную data-driven framework-систему
- делать редактор уровней
- строить backend
- полировать графику до завершения flow

---

## 25. Основной технический принцип проекта

Главный принцип этого этапа:

**сначала рабочий и чистый vertical slice, потом расширение**

То есть проект должен развиваться в порядке:

1. рабочий каркас
2. рабочий первый уровень
3. базовые системы
4. чистка кода
5. расширение на следующие главы
6. улучшение контента и подачи

## 27. Краткое резюме

- маленьким
